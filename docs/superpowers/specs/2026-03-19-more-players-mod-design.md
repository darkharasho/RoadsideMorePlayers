# RoadsideMorePlayers — Design Spec

## Overview

BepInEx 6 IL2CPP plugin for Roadside Research that increases the maximum player count beyond the default 4-player limit. Configurable, host-only enforcement for network limits, with a client-side bypass for the game's player count validation.

## Game Technical Profile

- **Engine**: Unity IL2CPP (HDRP)
- **Developer**: CyberneticWalrus
- **Steam App ID**: 3643170
- **Game code assembly**: `WallyBox`
- **Networking**: Photon Fusion 2 (`Fusion.Runtime`, `Fusion.Unity`, `Fusion.Realtime`)
- **Steam integration**: Steamworks.NET (`com.rlabrecque.steamworks.net`)
- **Backend**: PlayFab
- **Voice chat**: Dissonance + FMOD
- **Mod framework**: BepInEx 6.0.0-be.752 (Unity IL2CPP, win-x64)

## Patch Targets

### Patch 1: Photon Fusion Session Limit (Host-Only)

- **Target**: `Fusion.NetworkRunner.StartGame(Fusion.StartGameArgs)`
- **Assembly**: `Fusion.Runtime`
- **Patch type**: Prefix
- **Behavior**: Before `StartGame` executes, set `StartGameArgs.PlayerCount` to the configured max. Only applies when `GameMode` is `Host` or `Server`. The prefix must use `ref StartGameArgs` since it is a value type — modifications to the interop wrapper must propagate to the native struct.
- **Note**: Verify `StartGameArgs.PlayerCount` exists in the game's Fusion 2 SDK version from the interop assemblies. If absent, check for `SessionConfig` or equivalent.

### Patch 1b: Fusion Project Config Default (Host-Only)

- **Target**: `NetworkProjectConfig` — the `Simulation.DefaultPlayers` field
- **Assembly**: `Fusion.Runtime`
- **Patch type**: Runtime field override (set via `NetworkProjectConfig.Global` at plugin load)
- **Behavior**: Override `DefaultPlayers` to the configured max as a fallback. `StartGameArgs.PlayerCount` takes priority when set, but this ensures the default is also raised in case the game relies on the config elsewhere.

### Patch 2: Steam Lobby Size (Host-Only)

- **Target**: `Steamworks.SteamMatchmaking.CreateLobby(Steamworks.ELobbyType, int)`
- **Assembly**: `com.rlabrecque.steamworks.net`
- **Patch type**: Prefix
- **Behavior**: Override the `ref int cMaxMembers` parameter with the configured max. Preserve the original `ELobbyType` unchanged. This only fires on the host since only hosts create lobbies.

### Patch 3: Client-Side Player Count Validation (All Peers)

- **Target**: Method containing `LobbyCurrentPlayersMoreThanMaxPlayers` logic
- **Assembly**: `WallyBox` (exact class TBD — see Investigation Phase below)
- **Patch type**: Prefix returning `false` if the method returns `bool`, or postfix overriding the result
- **Behavior**: Override the check so it uses the configured max instead of the hardcoded value of 4.

### Patch 4: Host-Side Connection Rejection (Host-Only, if needed)

- **Target**: `INetworkRunnerCallbacks.OnConnectRequest` implementation in `WallyBox` (if present)
- **Assembly**: `WallyBox`
- **Patch type**: Prefix or Postfix
- **Behavior**: If the game implements `OnConnectRequest` to reject connections when `ActivePlayers.Count >= 4`, patch it to use the configured max. Only needed if this callback exists in the game code.

## Investigation Phase (Post-BepInEx Install)

These items require the IL2CPP interop assemblies generated by BepInEx's first run:

1. **Locate Patch 3 target**: Search `Il2CppWallyBox` for methods referencing `LobbyCurrentPlayersMoreThanMaxPlayers` or comparisons against the literal `4`
2. **Verify `StartGameArgs.PlayerCount`**: Confirm the field name and that IL2CPP struct patching propagates changes correctly
3. **Check for `OnConnectRequest`**: Search `WallyBox` for `INetworkRunnerCallbacks` implementations that enforce a player cap
4. **Audit `[Networked]` properties**: Check for fixed-size `NetworkArray` or `NetworkLinkedList` properties in `WallyBox` sized to 4 — if found, increasing player count beyond 4 could cause buffer overflows
5. **Check `NetworkProjectConfig.DefaultPlayers`**: Verify the baked default and whether runtime override works

## Configuration

File: `BepInEx/config/com.github.RoadsideMorePlayers.cfg`

```ini
[General]
## The maximum number of players allowed in a lobby (default 4 in vanilla)
## Only applies when you are the host
# Setting type: Int32
# Default value: 8
# Acceptable value range: From 1 to 32
MaxPlayers = 8
```

Minimum value is 2 (host + at least one client). Maximum 32.

## Host-Only Design

- Patches 1 and 2 only modify behavior when the local peer is the host/server
- Patch 3 runs on all peers to bypass client-side validation
- Clients without the mod can join expanded lobbies unless the game has additional client-side blocks beyond `LobbyCurrentPlayersMoreThanMaxPlayers`

## Known Limitations & Risks

- **All players should ideally have the mod** for the best experience, though only the host strictly needs it for networking
- **Fixed-size networked arrays** — If the game uses `[Networked, Capacity(4)] NetworkArray<T>` properties, extra players will cause out-of-bounds access in Fusion's networked state. This is the highest risk and must be investigated from the interop assemblies before declaring the mod viable.
- **High player counts (>16) may cause instability** — Photon Fusion sessions, Dissonance voice chat channels, game UI, and spawn point logic may not handle large counts gracefully
- **Spawn points** — The game likely has exactly 4 spawn points. Players 5+ may spawn at origin or cause null references. This may require a follow-up patch.
- **PlayFab session reporting** — If the game reports session player counts to PlayFab, exceeding 4 may trigger backend validation or anti-cheat flags
- **Exact Patch 3 and 4 method signatures** require IL2CPP interop assemblies from a first BepInEx run to identify
- **Game updates may break the mod** if CyberneticWalrus changes method signatures
- **UI limitations** — the game's lobby UI may not display more than 4 player slots; this mod does not modify UI
